# Security Guide

## Threat model

This is a fully static site: no server, no database, no user accounts, no
forms that submit data anywhere. That eliminates entire categories of
vulnerability (SQL injection, session hijacking, server-side request
forgery) by construction — there's no server-side code to exploit. What's
left to think about:

## 1. Supply chain (dependencies)

Every `npm install` pulls in code written by other people (Astro, Tailwind,
their transitive dependencies). A compromised package is the most realistic
risk to a project like this.

**Mitigations already in place:**
- `package-lock.json` (generated by `npm install`, committed to the repo)
  pins exact dependency versions — `npm ci` (used in the CI workflow)
  installs *exactly* what's locked, not "whatever's newest," so a
  dependency can't silently change under you between deploys.
- GitHub's **Dependabot** (free, enable it under repo **Settings →
  Security → Code security**) automatically opens pull requests when a
  dependency has a known vulnerability.

**What you should do periodically:** run `npm audit` (see
`docs/MAINTENANCE.md`) and review Dependabot PRs rather than ignoring them.

## 2. Cross-Site Scripting (XSS)

Astro's `{expression}` syntax **HTML-escapes by default** — any text
interpolated from your YAML content (link names, descriptions, etc.) is
automatically escaped, so even if malicious HTML somehow ended up in a
YAML file, it would render as inert text, not execute.

The one thing to actively avoid: **never use `set:html={...}`** with
content that isn't fully trusted and controlled by you. This directive
bypasses escaping and renders raw HTML. It is not used anywhere in this
codebase (checked deliberately during the rebuild) and should stay that way
unless you have a specific, reviewed reason to use it.

## 3. External links (`target="_blank"`)

Every external affiliate link uses `rel="noopener noreferrer"`. Without
`noopener`, a page you link to can use `window.opener` to redirect *your*
tab to a phishing page while its own tab stays open — a real, if
low-severity, attack pattern called "tab-nabbing." This is already handled
consistently across `SimpleCategory.astro`, `PlatformCategory.astro`, and
`JourneyCategory.astro`.

## 4. GitHub Actions workflow permissions

`.github/workflows/deploy.yml` requests the minimum permissions needed to
deploy to Pages (`contents: read`, `pages: write`, `id-token: write`) —
not broader repo-write access. If you add new workflows later, follow the
same principle: grant only what that specific job needs.

## 5. Secrets

This site currently needs **no secrets or API keys** — there's no
analytics, no backend, no third-party API calls at build or runtime. If you
add one in the future (e.g. a newsletter signup service), store it in
**GitHub Actions → Settings → Secrets**, never commit it to the repo, and
double-check it's not accidentally logged in a workflow run's output.

## 6. Content injection via YAML

Since category content lives in YAML files anyone with repo write access
can edit, the schema in `content.config.ts` acts as a light guard rail
(rejects malformed URLs, wrong types) but is not a security boundary — YAML
edits go through the same GitHub review process as code. If you ever accept
external contributions (see `docs/CONTRIBUTING.md`), review new/changed
links before merging, the same way you'd review a code change.
